# BrainScan — Метрики моделей

Контекст для Claude Code. Описывает как считаются метрики для двух моделей системы (локализатор области интереса и детектор опухоли) и end-to-end системы в целом.

---

## Базовый принцип

Обе модели — это задачи **детекции объектов** (возвращают bbox), а не классификации. Это меняет правила подсчёта метрик:

- **Считаем отдельно по предсказаниям** (каждое → TP или FP)
- **Считаем отдельно по эталонам** (каждый → TP или FN)
- Это две независимые матрицы счёта, одно изображение может породить и FP, и FN

**TN не определён** в задаче детекции. Невозможно подсчитать «сколько раз модель правильно ничего не нашла» — кандидатов на «не-объект» бесконечно много. Все стандартные метрики детекции (mAP, precision, recall, IoU) обходятся без TN.

**Порог IoU = 0.5** — стандартный baseline для всех расчётов (mAP@0.5).

---

## Модель 1: Локализатор области интереса

Задача: найти bbox области КТ/МРТ снимка на скриншоте экрана. На одном скриншоте может быть от 0 до N областей (мульти-монитор, несколько окон вьювера).

### Матрица счёта

```
TP (True Positive):
    эталонный bbox существует
    модель предсказала bbox
    IoU(предсказание, эталон) ≥ 0.5

FP (False Positive):
    модель предсказала bbox, и одно из двух:
    (a) на скриншоте эталонных bbox нет вообще, ИЛИ
    (b) предсказание не совпало ни с одним эталоном (IoU < 0.5 со всеми)

FN (False Negative):
    эталонный bbox существует
    модель его не нашла:
    (a) не предсказала ни одного bbox для этой области, ИЛИ
    (b) все предсказания имеют IoU < 0.5 с этим эталоном

TN: не применим
```

### Примеры для проверки понимания

```
Пример 1 (идеально):
- Эталон: 1 область
- Модель: 1 bbox, IoU = 0.8
→ TP = 1, FP = 0, FN = 0

Пример 2 (промах при наличии эталона):
- Эталон: 1 область
- Модель: 1 bbox, IoU = 0.3
→ TP = 0, FP = 1, FN = 1
  Один эталон порождает И FP, И FN одновременно:
  FP — за неверное предсказание, FN — за ненайденный эталон.

Пример 3 (галлюцинация):
- Эталон: 0 областей (чистый экран без снимка)
- Модель: 1 bbox
→ TP = 0, FP = 1, FN = 0

Пример 4 (правильное молчание):
- Эталон: 0 областей
- Модель: 0 предсказаний
→ TP = 0, FP = 0, FN = 0

Пример 5 (несколько эталонов):
- Эталон: 2 области
- Модель: 3 bbox: IoU=0.7 с областью 1,
                  IoU=0.6 с областью 2,
                  один — мимо всего
→ TP = 2, FP = 1, FN = 0

Пример 6 (частичный пропуск):
- Эталон: 3 области
- Модель: 1 bbox с IoU=0.9 на одну из них
→ TP = 1, FP = 0, FN = 2
```

### Метрики локализатора

```
Precision = TP / (TP + FP)
            Доля корректных bbox среди всех предсказаний модели.
            Низкий = модель часто рисует bbox не там.

Recall    = TP / (TP + FN)
            Доля найденных эталонных областей.
            Низкий = модель пропускает области.

F1        = 2 * Precision * Recall / (Precision + Recall)
            Гармоническое среднее, баланс между двумя метриками.

IoU_mean  = средний IoU только по TP-парам
            Качество найденных bbox.

mAP@0.5   = среднее precision по разным confidence thresholds
            при IoU threshold = 0.5
            Стандартная метрика детекции, агрегирует precision и recall.
```

### Приоритет метрик в медицинском контексте

**Recall — главная метрика локализатора.** Пропустить область со снимком (FN) означает: врач не увидит результата детекции опухоли вообще, потому что детектор даже не запустится на этой области.

Лишний bbox (FP) врач увидит и проигнорирует или исправит через UI. Это менее опасная ошибка.

## Модель 2: Детектор опухоли

Задача: найти bbox опухоли внутри уже локализованного crop'а области интереса. На вход crop, на выходе либо bbox опухоли, либо «опухоли нет».

### Особенность: появляется TN

В отличие от локализатора, **crop — это уже единица счёта**. Один crop = один бинарный исход «есть/нет опухоли». Это позволяет ввести TN на crop-level:

```
TP (crop-level):
    эталон сказал «есть опухоль» (есть аннотированный bbox опухоли)
    модель предсказала bbox опухоли
    IoU(предсказание, эталон) ≥ 0.5

FP (crop-level):
    модель предсказала bbox опухоли, и одно из двух:
    (a) эталон сказал «опухоли нет» (нет аннотированного bbox), ИЛИ
    (b) предсказание имеет IoU < 0.5 с эталонным bbox опухоли

FN (crop-level):
    эталон сказал «есть опухоль» (есть аннотированный bbox)
    модель её не нашла:
    (a) не предсказала ни одного bbox, ИЛИ
    (b) все предсказания имеют IoU < 0.5 с эталоном

TN (crop-level):
    эталон сказал «опухоли нет»
    модель не предсказала ничего
```

### Метрики детектора опухоли

```
Precision   = TP / (TP + FP)

Recall      = TP / (TP + FN)
              = Sensitivity в медицинской терминологии

Specificity = TN / (TN + FP)
              Доля чистых crop'ов правильно классифицированных как чистые.
              Критично для уменьшения ложных тревог.

F1          = 2 * Precision * Recall / (Precision + Recall)

AUC-ROC     = площадь под ROC-кривой, агрегатная метрика качества
              классификации при разных порогах confidence.

IoU_mean    = средний IoU по TP-парам, качество локализации опухоли.
```

### Приоритет метрик

**Sensitivity (Recall) — критичнейшая метрика.** Пропущенная опухоль (FN) — самая опасная ошибка в медицине. Любая опухоль которую модель не нашла — потенциальный вред пациенту.

**Specificity — вторая по важности.** Слишком много ложных тревог (низкий specificity) ведёт к alert fatigue — врачи начинают игнорировать предупреждения системы. Это косвенно увеличивает риск пропустить настоящую опухоль.

## End-to-End метрики системы

Система состоит из двух каскадных моделей: локализатор → детектор. Ошибки моделей **композируются**:

```
Локализатор: TP → Детектор: TP   ✓ опухоль правильно найдена
Локализатор: TP → Детектор: FN   ✗ область нашли, опухоль пропустили
Локализатор: FN → Детектор: --   ✗ область пропустили, до детектора не дошло
Локализатор: FP → Детектор: TP   ✗ нашли область там где её нет, «опухоль» — артефакт
Локализатор: FP → Детектор: TN   ⚠ ложная локализация, но детектор сказал «нет опухоли»
```

### Подсчёт end-to-end на уровне скриншота

Считаем на уровне скриншота, отвечая на вопрос «есть ли опухоль на этом скриншоте».

```
TP (system):
    на скриншоте была опухоль
    система обнаружила опухоль с правильным bbox
    (локализатор нашёл область + детектор нашёл опухоль внутри + bbox опухоли совпадает с эталоном)

FN (system):
    на скриншоте была опухоль
    система её не обнаружила (любая из ошибок в каскаде):
    - локализатор не нашёл область, ИЛИ
    - локализатор нашёл, но детектор пропустил опухоль, ИЛИ
    - детектор нашёл опухоль, но IoU с эталоном < 0.5

FP (system):
    на скриншоте опухоли не было (или не было области интереса)
    но система выдала bbox опухоли

TN (system):
    на скриншоте не было опухоли
    система не выдала ни одного bbox опухоли
```

### End-to-End метрики

```
Sensitivity = TP / (TP + FN)
              Доля скриншотов с опухолями, на которых система её обнаружила.
              Главная метрика безопасности.

Specificity = TN / (TN + FP)
              Доля чистых скриншотов правильно классифицированных как чистые.

NPV (Negative Predictive Value) = TN / (TN + FN)
              Если система сказала «опухоли нет», какова вероятность что её
              действительно нет. Метрика доверия к «зелёному» ответу системы.

PPV (Positive Predictive Value) = TP / (TP + FP) = Precision
              Если система сказала «опухоль есть», какова вероятность что
              она действительно есть.
```

**NPV особенно важен** для системы поддержки принятия решений: врач должен иметь возможность доверять отрицательному результату, иначе ценность скрининговой системы теряется.

## Скрытая проблема: bias в данных

Источник правды для метрик — **аннотация врача**, но врач размечает только то, что увидел и захотел проверить. Это создаёт **скрытые false negatives** на уровне сбора данных:

```
Сценарий риска:
- На экране была опухоль
- Врач не нажал «Detect» (не заметил, не интересовался этим снимком)
- Модель не запускалась
- Опухоль не размечена → не попадает в датасет → не учитывается в метриках
```

Это самая опасная категория ошибок в медицине: **unknown unknowns**. Метрики системы могут выглядеть отлично при том что реальный recall ниже расчётного.

Митигация:
- Периодический ревью случайной выборки скриншотов независимым экспертом
- Логирование всех скриншотов даже без действий пользователя
- Cross-validation против внешних размеченных датасетов (BraTS, TCIA) при дообучении

## Критерий продвижения новой версии модели в prod

### Для локализатора

Новая модель v(N) продвигается в prod если:

```
Recall(v_N)    >= Recall(v_prod)
Precision(v_N) >= Precision(v_prod) - 2%
```

Жёсткий floor по Recall — не допускаем регресс по самой важной метрике.
Допускаем небольшой регресс по Precision (false positives менее опасны и фильтруются врачом).

### Для детектора опухоли

Новая модель v(N) продвигается в prod если:

```
Sensitivity(v_N) >= Sensitivity(v_prod)
Specificity(v_N) >= Specificity(v_prod) - 2%
```

Аналогично: floor по Sensitivity (Recall), допустимый регресс по Specificity.

### Откат

Если после деплоя метрики на новых данных деградировали (например, sensitivity упала больше чем на 3% от заявленных на валидации) — автоматический откат на предыдущую prod-версию через таблицу `deployments`.

## Реализация подсчёта

### Где хранится IoU

Не денормализуем в схему. IoU вычисляется на лету при построении метрик путём сопоставления:

- `localize_detections.bbox` ↔ `localize_annotations.bbox` для локализатора
- `tumor_detections.bbox` ↔ `tumor_annotations.bbox` для детектора

Это позволяет переиграть метрики на исторических данных при смене порога IoU.

### Где хранятся итоговые метрики

В `models.metrics` JSONB на момент обучения:

```json
// для localize модели
{
  "iou_threshold": 0.5,
  "precision": 0.91,
  "recall": 0.89,
  "f1": 0.90,
  "iou_mean": 0.78,
  "map_50": 0.87,
  "test_dataset_id": "uuid..."
}

// для tumor модели
{
  "iou_threshold": 0.5,
  "precision": 0.93,
  "recall": 0.95,
  "f1": 0.94,
  "specificity": 0.92,
  "auc_roc": 0.97,
  "iou_mean": 0.81,
  "test_dataset_id": "uuid..."
}
```

### Мониторинг в проде

Помимо метрик на момент обучения, считаем live-метрики по производственным аннотациям:

```sql
-- За последние 7 дней: сравнение предсказаний модели с аннотациями врачей
SELECT
    DATE_TRUNC('day', la.annotated_at) AS day,
    la.action,
    COUNT(*) AS n
FROM localize_annotations la
WHERE la.annotated_at > now() - INTERVAL '7 days'
GROUP BY day, la.action;
```

Если `action = 'corrected'` встречается часто — модель деградирует на live-данных, нужно дообучение.
